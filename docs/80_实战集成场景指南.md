## 场景 1: 为新项目初始化开发环境

### 背景

新项目即将开始，需要为所有开发者设置统一的开发环境，包括：
- 特定版本的 Node.js (18.x) 和 Python (3.11)
- 自动环境加载 (.envrc)
- 预提交 Hook 检查
- 统一的 Git 配置

### 解决方案

#### 第一步：创建项目级配置

```bash
# 项目目录结构
my-project/
├── .mise.toml          # 版本管理
├── .envrc              # 环境变量
├── .pre-commit-config.yaml  # 预提交钩子
└── src/
    └── main.ts
```

#### 第二步：.mise.toml 配置

```toml
[env]
# 工具版本
node = "18.17.0"
python = "3.11.0"
rust = "1.72.0"

[tasks.dev]
run = "npm run dev"

[tasks.test]
run = "npm test"

[tasks.lint]
run = "npm run lint"

[tasks.format]
run = "npm run format"
```

#### 第三步：.envrc 配置

```bash
# 使用 Mise 管理的版本
use mise

# 项目特定变量
export PROJECT_NAME="my-project"
export NODE_ENV="development"
export API_URL="http://localhost:3000"

# 本地路径设置
export PATH_add="./node_modules/.bin"

# 可选：加载 .env.local（仅本地）
if [ -f .env.local ]; then
  dotenv .env.local
fi
```

#### 第四步：Git 配置

```bash
# 在项目目录执行
git config user.email "team@company.com"
git config core.safecrlf false

# 初始化 Lefthook
lefthook install

# 配置 pre-commit hooks
cat > lefthook-local.yml <<'EOF'
pre-commit:
  commands:
    lint:
      run: npm run lint
    test:
      run: npm test

commit-msg:
  commands:
    spell-check:
      run: npx commitlint --edit
EOF
```

### 预期流程

```bash
# 开发者克隆项目
git clone https://github.com/company/my-project.git
cd my-project

# Direnv 自动激活
# ✓ loading .envrc
# ✓ direnv: loading ~/projects/my-project/.envrc

# 所有工具版本自动设置
node --version   # 18.17.0
python --version # 3.11.0

# 开发
npm run dev

# 提交时自动运行检查
git commit -m "feat: new feature"
# ✓ 运行 lint
# ✓ 运行 test
# ✓ 运行 commitlint

# 如果检查失败，提交被阻止 ✓
```

## 场景 2: 多语言项目的版本管理

### 背景

维护一个包含多个语言的单体仓库（Monorepo）：
- Backend: Go 1.21
- Frontend: Node.js 18
- Data: Python 3.11
- CLI: Rust 1.72

需要：
- 按目录自动激活不同工具版本
- IDE 能感知版本变化
- CI/CD 使用相同的版本

### 解决方案

#### 项目结构

```
monorepo/
├── .mise.toml           # 全局版本配置
├── backend/
│   ├── go.mod
│   └── .mise.toml       # Backend特定版本（覆盖全局）
├── frontend/
│   ├── package.json
│   └── .mise.toml       # Frontend特定版本
├── data/
│   ├── requirements.txt
│   └── .envrc
└── cli/
    ├── Cargo.toml
    └── .mise.toml
```

#### 全局 .mise.toml

```toml
[env]
go = "1.21"
node = "18.17.0"
python = "3.11.0"
rust = "1.72.0"

[tasks.dev]
description = "Run all development servers"
run = """
echo "Starting development environment..."
direnv allow
"""

[tasks.test]
description = "Run all tests"
run = """
cd backend && go test ./...
cd ../frontend && npm test
cd ../data && pytest
cd ../cli && cargo test
"""

[tasks.build]
description = "Build all components"
run = """
cd backend && go build -o app
cd ../frontend && npm run build
cd ../data && pip install -e .
cd ../cli && cargo build --release
"""
```

#### Backend 特定 (.mise.toml)

```toml
[env]
go = "1.21.1"  # 覆盖全局版本
```

#### Frontend 特定 (.mise.toml)

```toml
[env]
node = "18.18.0"  # 覆盖全局版本
```

#### CI/CD 集成

```yaml
# .github/workflows/test.yml
name: Tests

on: [push, pull_request]

jobs:
  backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: jdx/mise-action@v2
      - run: cd backend && go test ./...

  frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: jdx/mise-action@v2
      - run: cd frontend && npm test

  data:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: jdx/mise-action@v2
      - run: cd data && pytest
```

### IDE 集成

**VS Code (.vscode/settings.json)**:

```json
{
  "go.lintOnSave": "package",
  "[javascript]": {
    "editor.defaultFormatter": "esbenp.prettier-vscode",
    "editor.formatOnSave": true
  },
  "[python]": {
    "editor.defaultFormatter": "ms-python.python",
    "editor.formatOnSave": true
  }
}
```

### 使用流程

```bash

cd backend
go version  # 1.21.1

cd ../frontend
node --version  # 18.18.0

cd ../data
python --version  # 3.11.0

# 一键运行所有测试

# ✓ Running backend tests

# ✓ Running data tests
```

## 场景 3: 跨团队的 Homeup 定制

### 背景

公司有三个团队，每个团队需要不同的工具和配置：

| 团队 | 工具需求 | 特殊配置 |
|------|---------|---------|
| **Backend** | Go, PostgreSQL, Redis | VPN + SSH tunneling |
| **Frontend** | Node, Vite, Tailwind | Figma token + Design tokens |
| **DevOps** | Terraform, Kubernetes, Docker | AWS credentials + Helm |

### 解决方案

#### 团队 Fork 和定制

```bash
# 组织结构

├── homeup (原始仓库)
├── homeup-backend (fork for backend team)
├── homeup-frontend (fork for frontend team)
```

## 场景 4: CI/CD 流程集成

### 背景

需要在 CI/CD 流程中复用 Homeup 的环境配置，确保：
- CI 环境与本地开发环境一致
- 快速构建（复用缓存）
- 可重现的构建

### 解决方案

#### GitHub Actions 集成

```yaml
# .github/workflows/ci.yml
name: CI

on: [push, pull_request]

jobs:
  setup:
    runs-on: ubuntu-latest
    steps:
      # 1. 使用 Mise Action
      - uses: jdx/mise-action@v2

      # 2. 缓存 Mise 安装
      - uses: actions/cache@v3
        with:
          path: ~/.local/share/mise
          key: mise-${{ runner.os }}-${{ hashFiles('.mise.toml') }}

  lint:
    needs: setup
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: jdx/mise-action@v2
      - run: npm run lint

  test:
    needs: setup
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: jdx/mise-action@v2
      - run: npm test
      - uses: codecov/codecov-action@v3

  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: jdx/mise-action@v2
      - run: npm run build
      - uses: actions/upload-artifact@v3
        with:
          name: build-artifacts
          path: dist/
```

#### Docker 集成

```dockerfile
FROM ubuntu:22.04

# 安装必要的工具
RUN apt-get update && apt-get install -y \
    curl git build-essential

# 安装 Homebrew
RUN /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# 克隆 Homeup
WORKDIR /root
RUN git clone https://github.com/company/homeup.git && \
    cd homeup && \
    just bootstrap

# 应用 Chezmoi 配置
RUN chezmoi init --source /root/homeup --apply

# 工作目录
WORKDIR /app

# 默认命令
CMD ["/bin/bash"]
```

#### Justfile 集成

```just
# justfile
deploy-ci:
    #!/bin/bash
    set -eo pipefail

    echo "Running CI pipeline..."

    # 1. 检查环境
    just doctor

    # 2. 验证依赖
    just validate-all

    # 3. 运行测试
    just test

    # 4. 构建
    just build

    # 5. 部署准备
    just pre-deploy-checks

pre-deploy-checks:
    #!/bin/bash
    echo "Checking deployment readiness..."

    # 验证所有包已安装
    brew bundle check

    # 检查环境变量
    [ -n "$DEPLOY_KEY" ] || (echo "DEPLOY_KEY not set"; exit 1)

    echo "✓ Ready for deployment"
```

### 缓存优化

```yaml
name: CI with Caching

on: [push, pull_request]

env:
  MISE_CACHE_DIR: /tmp/mise-cache

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Mise 缓存
      - uses: actions/cache@v3
        with:
          path: ${{ env.MISE_CACHE_DIR }}
          key: mise-${{ runner.os }}-${{ hashFiles('.mise.toml') }}
          restore-keys: |
            mise-${{ runner.os }}-

      # npm 缓存
      - uses: actions/cache@v3
        with:
          path: ~/.npm
          key: npm-${{ runner.os }}-${{ hashFiles('package-lock.json') }}

      - uses: jdx/mise-action@v2
      - run: npm ci
      - run: npm test
```

## 场景 5: 远程团队工作流

### 背景

团队分布在多个时区，需要：
- 异步代码审阅
- 自动化检查替代同步讨论
- 清晰的代码历史
- 统一的开发环境

### 解决方案

#### 预提交检查自动化

```yaml
# lefthook.yml
version: "2"

pre-commit:
  parallel: true
  commands:
    lint:
      glob: "*.{js,ts,jsx,tsx}"
      run: eslint {staged_files}
      stage_fixed: true

    prettier:
      glob: "*.{js,ts,json,yaml}"
      run: prettier --write {staged_files}
      stage_fixed: true

    spell:
      glob: "*.md"
      run: npx cspell {staged_files}

commit-msg:
  commands:
    commitlint:
      run: npx commitlint --edit

pre-push:
  commands:
    test:
      run: npm test
      fail_text: "Tests failed. Push anyway? [y/n]"

    build:
      run: npm run build
      fail_text: "Build failed. Push anyway? [y/n]"
```

#### Pull Request 模板

```markdown
# PR 检查清单

## 📝 描述

说明您做了什么以及为什么。

## 📸 截图（如果适用）

## ✅ 检查清单

- [ ] 代码遵循项目风格
- [ ] 自测通过
- [ ] 更新了相关文档
- [ ] 没有破坏性更改
- [ ] 添加了测试（如果适用）

## 🔗 关联 Issue

修复 #123

## 💬 审阅者建议

@reviewer1 @reviewer2 这个需要特别关注吗？
```

#### 异步评论指南

```
Good Practice 🟢:
- "行5-10：这个逻辑可以简化为..."
- "考虑使用 X 而不是 Y，因为..."
- "很好的实现！我建议再考虑..."

Avoid 🔴:
- "这是错的"
- "为什么你这样做？"
- "不好"
```

## 场景 6: 代码质量和安全保证

### 背景

需要确保：
- 代码质量标准

### 解决方案

#### 完整的检查流程

```bash
# justfile 中的验证任务
validate-code:
    #!/bin/bash
    set -eo pipefail

    echo "🔍 Code Quality Checks"

    # Linting
    echo "📝 Linting..."
    eslint src/

    # Type checking
    echo "🔤 Type checking..."
    tsc --noEmit

    # Security
    echo "🔐 Security scan..."
    npm audit --production

    # Tests
    echo "🧪 Running tests..."
    jest --coverage

    # Build
    echo "🏗️ Building..."
    npm run build

    echo "✅ All checks passed!"
```

#### Lefthook 秘密检测

```yaml
# lefthook.yml
pre-commit:
  commands:
    secrets:
      glob: "*.{js,ts,json,env,yaml,yml}"
      run: |
        # 检查常见秘密模式
        if grep -E "password|secret|api_key|token|credential" {staged_files} | \
           grep -v -E "// password|example|placeholder"; then
          echo "❌ Potential secrets detected!"
          exit 1
        fi

    env-files:
      glob: ".env*"
      run: |
        # 确保 .env 文件不被提交
        if [ -f .env ]; then
          echo "❌ .env should not be committed!"
          exit 1
        fi
```

## 场景 7: 新人快速上手

### 背景

新员工入职，需要在1小时内：
- 克隆所有需要的仓库
- 配置开发环境
- 运行项目
- 理解工作流

### 解决方案

#### Onboarding 脚本

```bash
#!/bin/bash
# scripts/onboard.sh

set -eo pipefail

echo "🎉 Welcome to the Team!"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"

# 1. 检查前置条件
echo "✓ Checking prerequisites..."
for cmd in git brew; do
  if ! command -v "$cmd" &> /dev/null; then
    echo "❌ $cmd not installed. Please install it first."
    exit 1
  fi
done

# 2. 克隆仓库
echo "✓ Cloning repositories..."
mkdir -p ~/Projects
cd ~/Projects

for repo in homeup my-project docs; do
  if [ ! -d "$repo" ]; then
    git clone "https://github.com/company/$repo.git"
  fi
done

# 3. 设置 Homeup
echo "✓ Setting up Homeup..."
cd homeup
just bootstrap

# 4. 设置项目
echo "✓ Setting up projects..."
cd ../my-project
direnv allow
mise install

# 5. 验证
echo "✓ Verifying setup..."
just doctor

# 6. 打开文档
echo "✓ Opening documentation..."
open docs/00_快速导航指南.md

echo ""
echo "✅ Onboarding complete!"
echo ""
echo "Next steps:"
```

## 💡 通用最佳实践

### 跨所有场景的建议

1. **自动化优先**
   - 使用 Mise 管理版本
   - 用 Lefthook 自动检查
   - 用 Just 自动化任务

2. **文档先行**
   - 每个决策都要记录
   - 新工具要更新文档
   - 保持示例最新

3. **循序渐进**
   - 先本地测试
   - 再在小团队试点
   - 最后逐步推行

4. **收集反馈**
   - 定期调查
   - 监听问题
   - 及时迭代

5. **支持新人**
   - 清晰的 onboarding
   - 易于查找的文档
   - 实时帮助渠道
